## 一、设计目标

1. **提示语始终可见**：用户输入的合并提示语不应被挤到页面下方
2. **流程清晰**：分步骤引导用户完成操作
3. **支持回退**：每一步都可以返回上一步修改
4. **单页处理**：支持对单个页面进行润色/扩展/改写/提取操作

## 二、页面结构

### 2.1 步骤指示器（顶部固定）

```
┌─────────────────────────────────────────────────────────────────┐
│  ① 上传 PPT  ──────→  ② 合并设置  ──────→  ③ 完成下载          │
│     (可点击)              (当前)              (未激活)           │
└─────────────────────────────────────────────────────────────────┘
```

**交互规则**：
- 已完成的步骤显示为 ✓ 可点击
- 当前步骤高亮显示
- 未激活步骤灰色禁用
- 点击已完成的步骤可回退

## 三、融合方式说明

| 模式 | 说明 | 提示语示例 |
|------|------|-----------|
| 整体合并 | 两份 PPT 全部内容融合 | "保留 A 的教学逻辑，补充 B 的例题" |
| 选择页面 | 用户选中部分页面融合 | "将选中的页面合并为一个完整单元" |
| 单页处理 | 对单个页面进行 AI 处理 | "润色这段文字，使其更通俗易懂" |

## 四、状态管理

### 4.1 步骤状态

```typescript
type Step = 'upload' | 'merge' | 'confirm'

interface MergeState {
  currentStep: Step

  // Step 1: 上传
  pptA: File | null
  pptB: File | null
  pptAPages: PptPageData[]
  pptBPages: PptPageData[]
  sessionId: string | null

  // Step 2: 合并设置
  mergeMode: 'full' | 'partial' | 'single'
  globalPrompt: string
  selectedPagesA: number[]
  selectedPagesB: number[]
  mergePlan: MergePlan | null

  // Step 3: 完成
  downloadUrl: string | null
  fileName: string | null
}
```

### 4.2 步骤切换规则

```
upload → merge:
  - 条件：pptA 和 pptB 都已上传且解析成功
  - 动作：自动切换

merge → confirm:
  - 条件：用户点击"确认方案"
  - 动作：调用 generate-final API

任意步骤回退:
  - 保留已输入的数据
  - 清除后续步骤的结果
```

## 五、响应式设计

### 5.1 桌面端 (> 1024px)
- 左右布局：左侧预览，右侧方案
- 顶部控制条固定

### 5.2 平板端 (768px - 1024px)
- 上下布局：顶部控制条，中间预览，底部方案
- 方案面板可折叠

### 5.3 移动端 (< 768px)
- 全屏步骤切换
- 底部 Tab 切换预览/方案
- 控制条简化
